Database Schema
We will extend prisma/schema.prisma to include models for chat persistence, email caching, and calendar event caching.

prisma

model Conversation {
  id        String    @id @default(uuid())
  userId    String
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  title     String
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  messages  Message[]
  @@index([userId])
  @@map("conversation")
}
model Message {
  id             String       @id @default(uuid())
  conversationId String
  conversation   Conversation @relation(fields: [conversationId], references: [id], onDelete: Cascade)
  role           String       // "user" | "assistant" | "system"
  content        String
  timestamp      DateTime     @default(now())
  @@index([conversationId])
  @@map("message")
}
model GmailCache {
  id         String   @id @default(uuid())
  userId     String
  user       User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  gmailId    String   @unique
  threadId   String
  subject    String
  sender     String
  snippet    String
  receivedAt DateTime
  createdAt  DateTime @default(now())
  @@index([userId])
  @@index([receivedAt])
  @@map("gmail_messages")
}
model CalendarEvent {
  id        String   @id @default(uuid())
  userId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  eventId   String   @unique
  title     String
  startTime DateTime
  endTime   DateTime
  attendees Json     // Array of attendee email strings
  createdAt DateTime @default(now())
  @@index([userId])
  @@index([startTime])
  @@map("calendar_events")
}

IMPORTANT

Database Host Migration: Since you've configured Corsair setups (gmail, googlecalendar) on your Docker container instance in google-demo, we will migrate the Corsair sync database tables (corsair_integrations, corsair_accounts, corsair_entities, corsair_events) to your Neon PostgreSQL instance so everything runs out of a single cloud DB.

Open Questions
IMPORTANT

Sync Strategy: Should we manually copy integration data into gmail_messages and calendar_events inside our webhook controller, OR is it acceptable to bypass duplicate tables and directly search/list using Corsair's native .db API methods (e.g., corsair.withTenant(userId).gmail.db.messages.list()) which uses the underlying corsair_entities table?

Recommended: We implement both. We use Corsair's native .db client for fast retrieval & searches, but also populate GmailCache and CalendarEvent on webhook events for clean Prisma joins if needed.

NOTE

LLM Provider: We plan to use the Vercel AI SDK with Google Gemini (using the GOOGLE_GENERATIVE_AI_API_KEY from your .env). Let us know if you prefer OpenAI or another LLM model instead.

Proposed Changes & Steps
The execution is divided into 5 Phases and 25 detailed steps:

Phase 1: Environment & Schema Preparation
1. Step 1: Install Integration Dependencies
Install the required packages in 
package.json
:

corsair
@corsair-dev/gmail
@corsair-dev/googlecalendar
@corsair-dev/mcp
@ai-sdk/google
ai
2. Step 2: Update Prisma Schema
Edit prisma/schema.prisma to add:

Conversation, Message, GmailCache, CalendarEvent models
Relations linking to the User model.
3. Step 3: Run Database Migrations
Execute prisma migrate dev to create the new tables on Neon PostgreSQL and regenerate the Prisma Client.

4. Step 4: Configure Environments
Copy CORSAIR_KEK and GOOGLE_GENERATIVE_AI_API_KEY to .env in the main workspace directory.

5. Step 5: Establish Corsair Client Core
Create lib/corsair.ts to initialize and export the Corsair engine with:

gmail() and googlecalendar() plugins
multiTenancy: true (scoped by user session ID)
Shared database connection pool.
Phase 2: Webhooks & Sync Infrastructure
6. Step 6: Next.js Webhook Route Handler
Create app/api/webhooks/route.ts to handle incoming Corsair Pub/Sub webhooks. It will run processWebhook to sync Google integration changes.

7. Step 7: Webhook Data Mirroring logic
Write background logic inside the webhook handler to identify updated messages/events, parsing the webhook payload, and syncing them into the GmailCache and CalendarEvent tables.

8. Step 8: Webhook Validation & Securing
Add secret verification to ensure incoming webhook calls originate securely from Google/Corsair.

9. Step 9: Establish Server-Sent Events / SSE route
Create app/api/sync/events/route.ts to push real-time refresh commands to the client workspace whenever webhooks execute.

10. Step 10: Sync Tunnel and Testing
Configure local environment webhooks utilizing ngrok, executing a testing command to verify Google Pub/Sub signals hit the Next.js API.

Phase 3: AI Agent & MCP Layer
11. Step 11: Create MCP Route Handler
Create app/api/mcp/route.ts to bridge Express-like MCP router to Next.js route format so that external MCP clients can interact with the app.

12. Step 12: Configure Gemini Vercel AI SDK Client
Create lib/ai.ts importing @ai-sdk/google to structure standard LLM options using Gemini.

13. Step 13: Corsair Tool-to-SDK Converter
Build helper function in lib/ai-tools.ts converting Corsair's tool definitions from buildCorsairToolDefs directly into execution tools compatible with Vercel AI SDK.

14. Step 14: System Rules & Assistant Prompting
Define agent roles: instructing Gemini how to read email caches, look up calendar availability, write email drafts, and trigger calendar creations.

15. Step 15: Create Chat Route Endpoint
Write app/api/chat/route.ts accepting conversation threads, executing generateText with the mapped tools, and updating the database with new user & assistant message turns.
